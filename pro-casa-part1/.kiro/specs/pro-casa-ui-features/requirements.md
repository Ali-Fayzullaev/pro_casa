# Документ требований: UI для трёх бизнес-фич Pro Casa

## Введение

Данный документ описывает требования к реализации пользовательского интерфейса для трёх бизнес-функций платформы Pro Casa (pro.casa.kz), бэкенд которых уже реализован:

1. **Подписки** — страница управления подписками пользователей для администратора
2. **Заявки в банки** — форма подачи ипотечной заявки для брокера
3. **TradeIn** — кнопка и форма обмена квартиры в карточке объекта CRM

Фронтенд построен на Next.js (App Router) с использованием shadcn/ui, TanStack Query и Axios. Бэкенд — Express + Prisma (PostgreSQL).

## Глоссарий

- **Система** — фронтенд-приложение Pro Casa (Next.js)
- **Админ** — пользователь с ролью ADMIN
- **Брокер** — пользователь с ролью BROKER
- **Страница_Подписок** — страница `/dashboard/admin/subscriptions` для управления подписками
- **Таблица_Подписок** — табличный компонент отображения всех подписок
- **Форма_Подписки** — диалоговое окно создания/обновления подписки пользователя
- **Страница_Заявок** — страница `/dashboard/mortgage-applications` для просмотра ипотечных заявок
- **Форма_Заявки** — диалоговое окно создания ипотечной заявки в банк
- **Кнопка_TradeIn** — кнопка в карточке объекта CRM для инициации обмена квартиры
- **Форма_TradeIn** — диалоговое окно с полями для создания TradeIn-сделки
- **API_Клиент** — Axios-инстанс (`lib/api-client.ts`) для запросов к бэкенду
- **Боковое_Меню** — компонент `AppSidebar` с навигацией по разделам
- **План_Подписки** — тарифный план: FREE, BASIC, PRO, ENTERPRISE
- **Статус_Подписки** — состояние подписки: ACTIVE, EXPIRED, CANCELLED
- **Статус_Заявки** — состояние ипотечной заявки: DRAFT, SUBMITTED, REVIEWING, APPROVED, REJECTED, CANCELLED

## Требования

### Требование 1: Страница управления подписками (Админ)

**User Story:** Как администратор, я хочу видеть все подписки пользователей и управлять ими, чтобы контролировать доступ к функциям платформы.

#### Критерии приёмки

1. WHEN Админ переходит по маршруту `/dashboard/admin/subscriptions`, THE Страница_Подписок SHALL отобразить Таблицу_Подписок со всеми подписками, полученными через `GET /api/subscriptions`
2. THE Таблица_Подписок SHALL отображать для каждой подписки: имя пользователя, email, роль, План_Подписки, Статус_Подписки, дату начала, дату окончания и сумму
3. WHEN Админ нажимает кнопку «Назначить подписку», THE Система SHALL открыть Форму_Подписки с полями: выбор пользователя, План_Подписки, дата окончания и сумма
4. WHEN Админ заполняет Форму_Подписки и нажимает «Сохранить», THE Система SHALL отправить запрос `POST /api/subscriptions` с данными формы и обновить Таблицу_Подписок
5. IF запрос `POST /api/subscriptions` возвращает ошибку, THEN THE Система SHALL отобразить уведомление об ошибке через toast-нотификацию
6. THE Боковое_Меню SHALL содержать пункт «Подписки» в разделе «Управление» только для роли ADMIN
7. WHILE данные подписок загружаются, THE Страница_Подписок SHALL отображать скелетон-заглушки вместо таблицы
8. THE Таблица_Подписок SHALL отображать План_Подписки с цветовым бейджем: FREE — серый, BASIC — синий, PRO — жёлтый, ENTERPRISE — зелёный
9. THE Таблица_Подписок SHALL отображать Статус_Подписки с цветовым бейджем: ACTIVE — зелёный, EXPIRED — красный, CANCELLED — серый

### Требование 2: Форма подачи ипотечной заявки (Брокер)

**User Story:** Как брокер, я хочу подавать ипотечные заявки в банки для клиентов, чтобы ускорить процесс получения ипотеки.

#### Критерии приёмки

1. WHEN Брокер переходит по маршруту `/dashboard/mortgage-applications`, THE Страница_Заявок SHALL отобразить список всех ипотечных заявок брокера, полученных через `GET /api/mortgage-applications`
2. THE Страница_Заявок SHALL отображать для каждой заявки: имя клиента, банк, программу, сумму кредита, срок, ставку, Статус_Заявки и дату создания
3. WHEN Брокер нажимает кнопку «Новая заявка», THE Система SHALL открыть Форму_Заявки с полями: выбор клиента (из списка `GET /api/clients`), название банка, название программы, сумма кредита, срок в месяцах и процентная ставка
4. WHEN Брокер заполняет Форму_Заявки и нажимает «Отправить», THE Система SHALL отправить запрос `POST /api/mortgage-applications` и обновить список заявок
5. IF поле «клиент» или «банк» или «сумма кредита» или «срок» не заполнено, THEN THE Форма_Заявки SHALL отобразить валидационное сообщение у соответствующего поля и заблокировать отправку
6. THE Страница_Заявок SHALL отображать Статус_Заявки с цветовым бейджем: DRAFT — серый, SUBMITTED — синий, REVIEWING — жёлтый, APPROVED — зелёный, REJECTED — красный, CANCELLED — серый
7. WHEN Админ или Брокер нажимает на строку заявки, THE Система SHALL позволить изменить Статус_Заявки через `PUT /api/mortgage-applications/:id/status`
8. THE Боковое_Меню SHALL содержать пункт «Заявки в банки» в разделе «Ипотека» для ролей ADMIN, BROKER и REALTOR
9. WHILE данные заявок загружаются, THE Страница_Заявок SHALL отображать скелетон-заглушки

### Требование 3: TradeIn — обмен квартиры в карточке объекта CRM

**User Story:** Как брокер, я хочу инициировать TradeIn-сделку из карточки объекта CRM, чтобы оформить обмен старой квартиры клиента на новостройку.

#### Критерии приёмки

1. WHEN Брокер просматривает карточку объекта CRM на канбан-доске, THE Система SHALL отображать Кнопку_TradeIn на карточке объекта
2. WHEN Брокер нажимает Кнопку_TradeIn, THE Система SHALL открыть Форму_TradeIn с полями: выбор ЖК (проекта), цена новой квартиры, процент комиссии (по умолчанию 1.5%), выбор клиента и комментарий
3. WHEN Брокер заполняет Форму_TradeIn и нажимает «Создать сделку», THE Система SHALL отправить запрос `POST /api/deals/tradein` с данными формы
4. IF запрос `POST /api/deals/tradein` выполнен успешно, THEN THE Система SHALL отобразить toast-уведомление «TradeIn сделка создана» и закрыть Форму_TradeIn
5. IF поле «цена новой квартиры» не заполнено, THEN THE Форма_TradeIn SHALL отобразить валидационное сообщение и заблокировать отправку
6. THE Форма_TradeIn SHALL автоматически подставить ID текущего объекта CRM как `oldPropertyId` и ID продавца объекта как `sellerId`
7. THE Форма_TradeIn SHALL отображать расчёт комиссии в реальном времени: комиссия = цена × процент / 100, Casa Fee = комиссия × 20%
8. WHILE список проектов загружается в Форме_TradeIn, THE Система SHALL отображать индикатор загрузки в поле выбора ЖК
